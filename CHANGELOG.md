## [Unreleased]

### Added
- **Solo CQ POTA: cut numbers (N for 9), default on.** RST is now keyed in cut form on the air — `54N` instead of `549`, `5NN` instead of `599` — which is how POTA / contest ops actually send it. The dummy RST shown to the user, the post-QSO summary card, and the working log all display cut form too. The parser accepts both forms (`5NN`, `5N9`, `54N`, `549`) anywhere in a transmission, including buried inside run-together tokens like `URBNNUR`. The grader normalizes user input via `fromCutNumbers()` (N → 9) before edit-distance comparison so all forms grade as identical. Internal state stays in canonical digit form; conversion happens only at the wire / display boundaries. New helper module `src/lib/cqpota/cutNumbers.js` with `toCutNumbers` / `fromCutNumbers` — the digit↔letter maps are objects so adding T (0) or A (1) later is one line per direction.

### Fixed
- **Solo CQ POTA: station responses dropped/partial when sent while the operator was still keying on the paddle.** Per the K1EL spec (`data.pdf` p.3): *"Paddle input will always take priority and will interrupt serial data, automatically clearing WK3's serial input buffer. When a paddle break-in occurs, any additional serial data that arrives from the host will be processed, but will be ignored unless it is an immediate command."* Software cannot interrupt the operator. When the user paddled `?` to request a repeat, the K1EL was clearing its serial input buffer mid-paddle and any bytes we wrote during that window got eaten. Fix: new `whenIdle()` helper in `src/lib/wkusb.js` returns a promise that resolves the next time the device's busy flag flips to false (or immediately if already idle), driven by the BUSY bit on the status bytes we already parse in `processStatusByte`. `useSoloQso`'s `stationSend` is now async and `await wkusbWhenIdle()` before each `wkusbSendMessage()` — the station response slots in cleanly *after* the operator finishes their transmission instead of fighting the paddle for the buffer. Reverted all the bandaids: the call-doubling in `responder.js`, the `Set WK3 Mode` / `Set Mode Register` push in `connect()`, and the `clearBuffer()` call before sends. Connect stays minimal as it always was. Added `enqueueWrite` console logging in `src/lib/wkusb.js` (label, byte count, hex + ASCII, and a "write resolved" line) so future write-path issues can be diagnosed from the dev console.

### Fixed
- **Solo CQ POTA: two different callsigns keyed back-to-back at session start.** React 18 StrictMode fires mount effects twice in dev (mount → cleanup → mount), so `SoloMode`'s mount effect was calling `solo.startSession()` twice — fetching two pools and shifting two different contacts, both written to the K1EL before the user could react. The K1EL ended up with 12 bytes of "two callsigns" jammed into its buffer, and from there everything downstream (`?` repeats, exchanges) hit a corrupted device state and silently dropped. Fix: `useSoloQso` now ref-guards `startSession()` so a second call is a clean no-op (the guard is reset by `endSession()` and on session-start failure), and `SoloMode`'s mount effect no longer calls `endSession()` in its cleanup — that's only invoked from the explicit "End session" button now. The hook's wkusb listener subscription has its own cleanup effect that handles real unmount.
- **Solo CQ POTA: wkusb writes are now serialized through a single promise chain.** Previously each `sendChar` / `sendText` / `sendMessage` / `sendProsign` / `clearBuffer` called `writer.write()` directly, returning an unawaited promise. Two rapid sends (e.g. an initial CQ call followed by a `?` repeat 200ms later) had overlapping writes in flight, which Web Serial / the K1EL did not handle gracefully — bytes were dropped or interleaved. New module-level `enqueueWrite()` chains every write off the previous one's resolution so there's never more than one `writer.write()` outstanding at a time.

- **Solo CQ POTA: station state was hardcoded to `NE NE`.** The exchange reply now keys the contact's actual state from their HamDB lookup. `api/solo-pool.js` and `api/hamdb.js` now extract `state` from the HamDB JSON (sanitized to a 2-letter US abbreviation), the contact pool entries carry it through as `{ call, name, state }`, and `buildExchangeReply` takes a third `contactState` arg. If the lookup didn't return a state, the responder drops the `<state> <state>` segment entirely rather than fabricating one.
- **Solo CQ POTA: lenient parsing of run-together exchanges.** Real CW spacing is fluid — operators routinely smush words like `TU BRIAN UR 429` into a single emitted token `TUBRIANUR429`. The strict token-based parser was dropping those on the floor (no RST, no name, no TU credit). The parser now: returns *every* call-shaped token in `callAttempts` (instead of guessing the first), scans the raw upper text for `[1-5][1-5]9` substrings into `rstAttempts` (so RSTs buried inside smushed words are still found), exposes a `runTogetherTokens` list of mixed-alphanumeric exchange tokens, and uses that list to find `TU` / `BK` / `73` / `FB` markers buried inside the smush without false-positiving on pure-letter names like ARTHUR. The grader now picks the best call/RST candidate against the expected values via edit distance (so a "<own_call> <contact_call>" send order still grades against the contact), and the name bonus accepts the expected name as a substring inside any run-together token. `qsoMachine.hasExchange()` and `useSoloQso`'s callsign log-cell logic were updated to read the new arrays.

### Added
- **Solo: CQ POTA practice mode.** A single-player mode for when nobody else is online. Picks one of six US POTA parks per session, pulls the leaderboard + recent activations, hydrates each callsign with operator names, and walks the user through realistic POTA exchanges against the keyer. Implements the partial-call etiquette (`?`, `X?`, `QRZ?`, `<call>?`), the full RST exchange, and a closing transmission. Boundaries between user transmissions are detected from `BK`, trailing `?`, or 1.5s of input silence — paddle echoes from `wkusb.onChar` and keyboard input both flow through the same buffer. The calling station occasionally collapses `BK` into a Merge-Letters prosign (~5%) for flavor. Per-QSO grading covers callsign accuracy (heavy weight, partial credit via Levenshtein), RST accuracy, name bonus, time-of-day greeting (GM/GA/GE), TU/73 closers, signing the user's own call in the close, and an `FB` multiplier capped at 2.0x. Running session score and a working log of completed contacts persist across reloads via `localStorage` (`morseChat.solo.sessionScore`, `morseChat.solo.userName`).
- New endpoint `api/solo-pool.js` that does the entire session orchestration server-side: shuffles the park list, fetches POTA leaderboard + activations in parallel, hydrates the deduped callsign set against HamDB in parallel (capped at 15), drops misses, and rotates to the next park if too few survive. The browser pays a single round trip to start a session instead of fanning out 30+ requests through proxies. `api/pota.js` and `api/hamdb.js` remain for ad-hoc lookups (the user's own callsign still goes through `/api/hamdb`).
- New files: `api/pota.js`, `api/hamdb.js`, `api/solo-pool.js`, `src/lib/cqpota/{parks,callPool,parser,responder,grader,qsoMachine}.js`, `src/hooks/useSoloQso.js`, `src/components/SoloMode.jsx`. Wired into `App.jsx` as a third top-level branch alongside `ChatView` and the lobby; the lobby now shows a "Start solo" card next to the online toggle (disabled until WKUSB is connected, since the mode keys the calling station's transmissions on the hardware).
- `wkusb.sendProsign(a, b)` and `wkusb.sendMessage(parts)` helpers in `src/lib/wkusb.js`. `sendProsign` writes `0x1B XY` (Buffered Cmd: Merge Letters per K1EL WK3, page 17 of `data.pdf`) so two letters key as a single prosign with no inter-character space. `sendMessage` accepts a mixed array of plain strings and `{ prosign: 'XX' }` objects and serializes them into one buffered write so the responder can compose mixed text + prosign messages cleanly.
- Vercel Analytics: `@vercel/analytics` dependency, `<Analytics />` mounted in `src/main.jsx` alongside `<App />`. Enable in the Vercel project dashboard for the data to start flowing.

### Changed
- Lobby user list is now visible regardless of the user's own online state. The "Appear online" toggle still controls whether *you* show up to others, but you can browse who's online and initiate connection requests without first making yourself visible. `OnlineToggle` copy updated to reflect this.

### Fixed
- **`Set PinCfg` and `Set WinKeyer Mode` no longer clobber unrelated bits.** The K1EL protocol writes the entire byte for both registers and offers no register read-back, so the previous live setters were silently zeroing hang time, ultimatic priority, paddle swap, autospace, contest spacing, and the paddle watchdog disable bit on every toggle. The settings object now tracks the **full** state of both registers; `computeModeRegister()` and `computePinConfig()` build the byte from every field. Defaults mirror the working reference at `newqrzframe/public/wkusb-manager.js` (`defaultModeRegister = 0x50`, `defaultPinConfig = 0x06`); a default-state round-trip produces exactly those bytes. Bits 6 (paddle echoback ON) and 2 (serial echoback OFF) of the mode register remain forced because they're load-bearing for the chat→pair routing model.

### Added
- WKUSB settings panel (collapsible, below the connection bar) with sliders/toggles for WPM speed, **max speed (slider upper bound)**, **key mode (Iambic A/B/Ultimatic/Bug)**, sidetone on/off, sidetone frequency (500–1500 Hz), and PTT enable. Settings persist in `localStorage` (`morseChat.wkusbSettings`) and are applied live to the device when connected — or staged for next connect when offline.
- Public setter API on `src/lib/wkusb.js`: `setWpm`, `setMaxWpm`, `setSidetoneEnabled`, `setSidetoneHz`, `setKeyMode`, `setPttEnabled`, plus `getSettings` / `onSettings` for state subscription. Wire commands: WPM = Set Speed (`02 nn`), max WPM = Setup Speed Pot (`05 min range pad`), sidetone freq = Sidetone Control (`01 nn`, byte = 62500 / Hz), key mode = Set WinKeyer Mode (`0E nn`) with the mode-register field rebuilt from `MODE_REGISTER_BASE | KEY_MODE_BITS[mode]`, sidetone enable + PTT enable rebuild PinCfg from `PIN_CONFIG_BASE | flags` and write Set PinCfg (`09 nn`).

### Removed
- **Sidetone volume control.** The Admin Set Volume command (`00 24 nn`) was breaking connection on the user's hardware. Setting, setter, hook export, panel UI, and the post-Load-Defaults volume write are all gone.
- **All init defaults pushed at connect time.** The `connect()` bring-up no longer sends `Set WK3 Mode` (`00 14`) nor the `Load Defaults` block (`0F` + 15 values). After the host-open handshake the device retains whatever settings it had standalone. User-set values are only pushed when the user actively changes a control in the settings panel — that path uses the focused per-setting commands which are known-safe.

### Notes
- Setting max WPM clamps the current WPM downward if it exceeded the new max, then pushes both Setup Speed Pot and Set Speed in one go so the device and slider stay in sync.

### Changed
- Branded the app as "somber's morse chat" (browser tab title and in-app header).
- `src/lib/wkusb.js` is no longer a stub: real Web Serial integration for WKUSB / WKMini / WKUSB-AF using the K1EL WinKeyer 3 protocol. Connect at 1200 8N1, host-open handshake (`00 02`) with revision-code discard, then `00 14` (Set WK3 Mode) followed by Load Defaults (`0F` + 15 values). Mode register `0x50` enables paddle echoback and disables serial echoback so we never receive our own writes back. Read loop classifies bytes by tag (status / speed-pot / echo) and dispatches paddle-sourced ASCII to `onChar` listeners. Adds `onState`, `clearBuffer`, and `isBusy` to the public API.
- ChatView now plays remote `morse-text` on the local WKUSB (so the operator hears their partner's keying), and locally typed/pasted characters also go to the WKUSB so they're heard/transmitted alongside being published. Paddle-sourced characters are sent to the pair channel but never looped back to the device.
